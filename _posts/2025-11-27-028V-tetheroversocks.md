## 서론
[이 포스트](/server/2024/09/17/RE5012-TetherAPN.html)에서 통신사의 테더링 제한을 우회하는 방법을 설명했었다.<br>
그러나 저 설정 방법은 자급제 안드로이드 기기(외산 포함)가 아닌 단말기에선<br> 
구현할 수가 없다는 한계가 있었다.<br>

물론 저렇게 APN을 수정하지 않아도 기기 안에서 프록시 서버를 열어<br> 
테더링의 트래픽을 기기의 트래픽으로 만드는 방법은 여럿 존재한다.<br>
안드로이드에선 PdaNet+와 Netshare라는 앱으로 해당 기능을 구현할 수 있다.<br>
아이폰에서는 iSH라는 유료 앱에서 아래의 명령을 입력하면 유사하게 구현이 가능했다.<br>
```plaintext
apk update && apk add tinyproxy && sed -i 's/^Allow 127.0.0.1$/Allow 0.0.0.0\/0/' /etc/tinyproxy/tinyproxy.conf && echo -e "tinyproxy &\ncat /dev/location > /dev/null &\nps aux | grep tinyproxy | grep -v grep\ngrep '^Allow' /etc/tinyproxy/tinyproxy.conf" >> ~/.profile && source ~/.profile
```

클라이언트 단에서 추가적으로 프록시 설정을 해줘야 하는 것에 대해선 설명을 생략한다.<br>
아무튼 이렇게 되면 웹 브라우징 같은 건 잘 되는데 뭔가 안되는 게 하나 둘 보일 것이다.<br>
카카오톡이나 디스코드, 또 UDP 통신을 쓰는 다양한 서비스들이 정상적으로 작동하지 않을 것이다.<br>
이는 프록시 서버가 TCP 패킷만을 넘겨서 그런 것이며, <br>
TCP 통신이 아닌 모든 통신은 통신사가 설정한 공유데이터를 차감시키거나,<br>
없다면 되지 않을 것이다.<br>

## 방법이 없을까?
많은 사람들이 이 대목에서 피로감을 느끼고 포기를 한다.<br>
인터넷이 되기는 한다는 것에 만족하고 그냥 쓰거나,<br>
이에 만족하지 못한다면 라우터를 추가로 구입하거나,<br>
또는 셀룰러 모델로의 교체를 하는 식으로 말이다.<br>
나 또한 아이폰을 주력으로 쓰게 된 이후로는 위 2, 3번째 선지를 계속 만지작거렸다.<br>

경험상 전자보다는 후자가 더 쾌적했다.<br>
전자는 중국제 라우터가 시장을 잠식한지 오랜 시간이 지났으며,<br>
그 성능과 사용시간에 신뢰성을 느끼기가 어려웠다.<br>
더구나 추가적인 단말기라는 측면에서<br>
외부에서는 미니멀리즘을 지향하는 내 취향에 맞지 않았다.<br>
카페에 바리바리 싸들고 와서 전개하는 건 성가신 일이다.<br>
그런 걸 하기엔 집이라는 좋은 장소가 있다!<br>
후자는 다 좋은데 기기가 너무 비싸고,<br>
무엇보다 멀쩡히 쓰는 폰 회선을 냅두고<br>
높은 요금제의 추가적인 개설을 요한다.<br>
메인 회선을 저렴하게 바꾸고 기기에 몰아주는 방법도 있지만,<br>
영상을 많이 보고 싶은 사람이면 별로 내키지 않을 것이다.<br>

### UDP over TCP
찾아보니 TCP 터널 안에 UDP 패킷을 집어넣는 것이 가능은 하다고 한다.<br>
UDP의 이점을 상당부분 훼손하는 바보같은 방법이지만 그래도 써야만 한다면...<br>
나의 경우 UDP 통신을 활용하는 Wireguard VPN과 Moonlight가 필수적이기 때문에<br>
이 부분은 나의 구미를 당겼다.<br>

그러나 이를 위해서는 서버와 클라이언트에 모두<br> 
UDP over TCP가 가능하도록 환경을 구성해야 한다.<br>
지금은 아이폰이 라우터고, 거기에 프록시 서버까지 돌리고 있는데<br>
거기에 추가적으로 해당 구성이 가능해보이지 않았다.<br>
서버를 돌릴 앱인 iSH가 Alpine 리눅스를 가상으로 돌려 기능을 제공하는데,<br>
문제는 네트워크 어댑터 관련 권한 문제가 있어<br>
프록시 서버 주소조차 특정을 할 수가 없다.<br>
때려 맞춘 주소로만 TCP 통신이 가능하다.<br>
일반적으로 접속을 시도할 때도 이럴진데,<br>
사전에 주소와 포트를 지정하고 UDP 패킷이 이 터널을 지나도록<br>
구성하는 것은 난도를 산으로 올려버리는 짓 같았다.<br>

### Socks5?
Socks는 Socket Secure의 약자로 전송계층(L4)과 응용계층(L7)에서 작동한다.<br>
HTTPS와는 다르게 내부에 어떤 패킷이 들어오든 서버 <-> 클라이언트로 전달하며<br>
최근 규격인 Socks5에서는 UDP 패킷과 인증을 지원한다.<br>
이를 이용하여 클라이언트 발 Wireguard에 감싸진 Moonlight의 UDP 패킷을<br>
프록시 서버를 통해 정상적으로 전달할 수 있다.

## 클라이언트 설정
저렇게 연결하기 위해선 클라이언트에서도 Socks 프록시 서버에 접속할 수 있어야 한다.<br>
아래에 있는 주소는 동일하지 않을 수 있고, 실행 화면에 뜨는 걸 기준으로 입력해야한다.<br>
### iOS의 경우
HTTP 프록시 설정에서 PAC 주소인 http://172.20.10.1:8088/wpad.dat을 입력하여 접속한다.<br>
### macOS의 경우
System Preferences를 열고 Network, Wi-Fi, Advanced 순서대로 들어가 프록시 설정에서 SOCKS Proxy를 체크한다.<br>
주소를 입력하는 부분에 http://172.20.10.1:9876을 치고 접속한다.<br>
### Windows와 Linux경우
시스템에서 제공하는 프록시 서버 설정 단에서 PAC 주소인 http://172.20.10.1:8088/wpad.dat을 입력하여 접속한다.
### Android의 경우
Socks 클라이언트 역할을 해줄 앱으로 http://172.20.10.1:9876에 접속한다.<br>
인증 정보가 있다면 Shadowsocks 앱도 괜찮지만,<br>
나는 Tun2Socks 앱을 사용했다.<br>
+ GitHub 설명에서는 Socks 프록시를 지원하지 않기 때문에 HTTP 프록시로 연결하라 했지만,<br>
이처럼 VPN 슬롯을 통해서라도 Socks 프록시로 연결하는 방법이 존재한다.<br>
